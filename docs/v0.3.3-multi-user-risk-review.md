# 마찰지도 v0.3.3 다중 사용자 리스크 리뷰

작성일: 2026-04-30  
대상 버전: v0.3.2 감성 프리미엄 UI 리디자인 이후  
목적: 자동 QA로 확인하기 어려운 실제 사용자 사용성 위험을 정리하고, Google Play 출시 전 검증 우선순위를 정한다.

## 전제

v0.3.2 자동 시뮬레이션 QA는 기능 안정성 관점에서 의미 있는 기준을 통과했다. 다만 자동 테스트 통과는 기능 안정성 통과이지, 실제 사용성 통과는 아니다.

자동 QA는 버튼이 눌리는지, localStorage에 저장되는지, 화면 전환이 되는지, 데이터가 새로고침 뒤 유지되는지를 확인할 수 있다. 그러나 처음 사용자가 앱의 목적을 바로 이해하는지, 표현이 부담스럽지 않은지, 손가락으로 편하게 누를 수 있는지, 며칠 뒤 다시 돌아와 쓸 이유가 생기는지는 실제 사용자 테스트로 확인해야 한다.

마찰지도는 치료나 진단 도구가 아니라, 자꾸 막히는 순간을 한 줄로 남기고 반복되는 패턴을 보는 개인 기록 앱이다. 따라서 실제 사용자 검증에서도 “효과를 보장한다”는 관점이 아니라, 사용자가 자신의 기록을 편하게 남기고 살펴볼 수 있는지를 확인한다.

## 1. 자동 QA에서 통과한 항목 요약

자동 시뮬레이션에서 확인한 항목은 다음과 같다.

| 영역 | 통과 내용 |
| --- | --- |
| 첫 화면 | 앱 목적, 핵심 문구, 하단 탭 표시 확인 |
| 빈 상태 | 샘플 미리보기와 실제 저장 아님 안내 확인 |
| 기록 입력 | 빈 입력 방지, 예시 칩 입력 보조, 기록 저장 확인 |
| 예시 칩 | 텍스트, 감정, 생활 영역, 마찰 단계, 강도 기본값 반영 확인 |
| 저장 유지 | 새로고침 후 기록 유지 확인 |
| 지도 | Top 3, 범례, 영역/단계 표시 확인 |
| 시도 카드 | 생성, 기본 상태, 상태 변경, 새로고침 유지 확인 |
| 연결 안정성 | 연결된 기록 삭제 후에도 시도 카드 유지 확인 |
| 기록 관리 | JSON 내보내기, 시도 카드 삭제, 모든 기록 삭제 확인 |
| 다크모드 | 토글과 새로고침 후 유지 확인 |
| 콘솔 | error/warning 없음 확인 |
| 모바일 폭 | 주요 화면에서 페이지 단위 가로 overflow 없음 확인 |

추가로, 모바일 스크롤 상태에서 하단 탭 클릭이 본문 카드에 가로막힐 수 있는 문제가 발견되어 탭 레이어를 보강했다.

## 2. 자동 QA로는 확인할 수 없는 위험

### 2.1 첫 기록까지의 실제 심리적 부담

자동 QA는 사용자가 45초 안에 기록을 남길 수 있는지 기술적으로만 확인한다. 실제 사용자는 “무엇을 적어야 하지?”, “이걸 적어도 되나?”, “분류를 정확히 골라야 하나?” 같은 망설임을 겪을 수 있다.

확인해야 할 질문:

- 첫 화면만 보고 무엇을 하는 앱인지 이해하는가?
- “막힌 순간”이라는 표현이 충분히 친숙한가?
- 예시 칩이 실제 기록이 아니라 입력 도움이라는 점을 이해하는가?
- 감정/영역/단계 선택이 정답 고르기처럼 느껴지지 않는가?

### 2.2 지도 화면 해석 부담

지도 화면은 앱의 핵심 차별점이지만, 처음 보는 사용자는 히트맵을 표처럼 느낄 수 있다. Top 3와 쉬운 해석 문장이 없으면 “그래서 뭘 보면 되지?”라는 반응이 나올 수 있다.

위험:

- 생활 영역과 마찰 단계의 조합 개념이 바로 이해되지 않을 수 있다.
- 숫자와 색상 강도가 충분히 설명되지 않으면 지도처럼 느껴지지 않을 수 있다.
- 기록이 1~2개뿐일 때 사용자가 성급하게 패턴으로 받아들일 수 있다.

필요한 보강:

- Top 3는 지도 위쪽에 계속 유지한다.
- “아직 관찰을 시작한 단계예요” 같은 기록 수별 안내를 유지한다.
- 지도 아래에 “바로 고치지 않아도 괜찮습니다”처럼 부담을 낮추는 문구를 둔다.

### 2.3 시도 카드 입력 부담

시도 카드는 앱을 행동 자기조절 루프로 확장하는 핵심이지만, 입력 항목이 많으면 숙제처럼 느껴질 수 있다. 자동 QA는 모든 필드를 채울 수 있는지만 확인할 뿐, 사용자가 그 과정을 부담 없이 받아들이는지는 확인하지 못한다.

위험:

- “왜 그랬을까요?”가 추궁처럼 느껴질 수 있다.
- “어떻게 확인할까요?”가 너무 분석적으로 느껴질 수 있다.
- 5개 이상의 입력 필드가 모바일에서 길게 느껴질 수 있다.
- 시도 카드를 만들기 전에 기록을 더 모아도 된다는 안내가 약하면 사용자가 서둘러야 한다고 느낄 수 있다.

필요한 보강:

- 기본 예시 문장을 제공하되 정답처럼 보이지 않게 한다.
- 선택 입력과 필수 입력을 다시 나눈다.
- 첫 버전에서는 “시도 카드 만들기”보다 “작게 해볼 방법 정하기” 같은 표현을 검토한다.

### 2.4 돌아보기 루프의 약함

현재 구조는 기록과 시도 카드 생성은 가능하지만, 장기 사용 관점에서는 “해보니 어땠는지”를 돌아보고 다음 시도를 조정하는 루프가 더 강해야 한다.

위험:

- 사용자가 시도 카드를 만들고 다시 돌아오지 않을 수 있다.
- 상태 변경이 “해보는 중”에서 끝날 수 있다.
- 돌아보기 입력이 실제 다음 시도로 이어지지 않을 수 있다.

필요한 보강:

- “살펴봄” 상태가 다음 작은 조정으로 이어지도록 한다.
- 돌아보기에서 “다음에는 더 작게 해볼까요?” 같은 연결 문구를 둔다.
- 시도 카드 복제 또는 더 작은 시도 만들기 기능은 다음 버전 후보로 둔다.

### 2.5 Android 실제 기기 변수

자동 QA는 모바일 viewport를 흉내 내지만, 실제 Android WebView, 키보드, 뒤로가기, 안전 영역, 화면 비율은 다를 수 있다.

특히 반드시 확인해야 할 항목:

- Android 실제 기기에서 키보드와 하단 탭 충돌
- textarea 입력 중 하단 탭이 가려지거나 입력창을 밀어내는 방식
- 하단 탭이 본문에 가려지지 않는지
- 긴 카드와 작은 화면에서 버튼이 손가락으로 누르기 쉬운지
- Android 뒤로가기 버튼을 눌렀을 때 앱이 이상하게 종료되지 않는지

### 2.6 개인정보 신뢰

마찰지도는 사용자가 사적인 내용을 적을 수 있는 앱이다. 앱이 로컬 저장을 사용하더라도, 사용자는 “내 기록이 어디로 가는지”를 명확히 알고 싶어 한다.

위험:

- 개인정보처리방침 URL이 자리 표시 상태라면 신뢰가 낮아진다.
- 문의 이메일이 실제 값이 아니면 Play Store 심사와 사용자 신뢰에 불리하다.
- 기록 내보내기와 모든 기록 삭제가 있어도, 처음 사용자가 그 기능을 발견하지 못할 수 있다.

필수 보강:

- 개인정보처리방침 URL을 실제 공개 주소로 연결한다.
- 문의 이메일을 실제 사용 가능한 주소로 넣는다.
- “기록은 이 기기 안에 저장됩니다” 문구를 돌아보기 화면 하단뿐 아니라 온보딩/스토어 설명에도 일관되게 사용한다.

### 2.7 장기 사용성

첫날 기록은 가능해도, 사용자가 3일, 7일, 14일 뒤에도 돌아올지는 별개의 문제다.

위험:

- 기록이 쌓이지 않으면 지도 가치가 늦게 온다.
- 시도 카드를 만들고 나면 다음 행동이 약해질 수 있다.
- 알림을 넣지 않는 철학은 좋지만, 재방문 동기가 약할 수 있다.

보강 방향:

- 알림 대신 앱 안에서 “오늘 하나만 남겨도 충분합니다”라는 낮은 진입 문구를 유지한다.
- 돌아보기 화면에서 최근 변화나 누적 기록 수를 조용히 보여준다.
- streak나 점수 없이도 “관찰이 이어지고 있다”는 느낌을 줄 방법을 찾는다.

## 3. 다양한 사용자 유형별 예상 문제

### 자기관리 앱에 익숙한 사용자

예상 반응:

- 체크리스트, 목표, 달성률, 통계 대시보드를 기대할 수 있다.
- 점수나 streak가 없어서 처음에는 “뭘 잘하고 있는지” 모르겠다고 느낄 수 있다.

위험:

- 앱이 너무 느슨하게 느껴질 수 있다.
- 시도 카드가 목표 관리 기능처럼 보이지 않아 혼란이 생길 수 있다.

대응:

- “할 일을 더 늘리지 않는다”는 철학을 유지한다.
- 지도와 Top 3를 앱의 가치로 더 선명하게 보여준다.
- 달성률 대신 “자주 막힌 위치”와 “다음에 덜 버겁게 할 방법”을 강조한다.

### 자기관리 앱을 잘 안 쓰는 사용자

예상 반응:

- 분류 선택이 복잡하다고 느낄 수 있다.
- “마찰”, “단계”, “지도” 같은 개념이 처음에는 낯설 수 있다.

위험:

- 첫 기록 전에 이탈할 수 있다.
- 감정/영역/단계를 정확히 골라야 한다고 오해할 수 있다.

대응:

- 예시 칩을 첫 화면 상단에 계속 유지한다.
- “정확히 고르지 않아도 괜찮습니다” 문구를 유지한다.
- 기본값이 자동으로 채워지는 경험을 더 자연스럽게 만든다.

### 바쁜 직장인

예상 반응:

- 빠르게 기록하고 앱을 닫고 싶어 할 가능성이 높다.
- 긴 시도 카드 입력은 나중으로 미룰 수 있다.

위험:

- 기록은 하지만 지도나 시도 화면까지 가지 않을 수 있다.
- 긴 입력 필드가 업무 도구처럼 느껴질 수 있다.

대응:

- 첫 기록은 30~45초 안에 끝나야 한다.
- 시도 카드는 “나중에 작게 바꿔보기” 흐름을 허용한다.
- 최근 기록 1~2개와 지도 진입 버튼을 가볍게 제공한다.

### 감정적으로 예민한 사용자

예상 반응:

- “왜 그랬을까요?”를 자기 비난처럼 받아들일 수 있다.
- 기록을 남기는 것 자체가 부담스럽게 느껴질 수 있다.

위험:

- 문구가 조금만 단정적으로 보여도 이탈할 수 있다.
- 지도나 Top 3를 자신에 대한 평가처럼 받아들일 수 있다.

대응:

- “진단”, “문제”, “나쁜 습관”, “성공/실패” 같은 표현을 계속 피한다.
- Top 3 옆에 “힌트”나 “관찰” 표현을 유지한다.
- 시도 카드에는 “바로 해결하지 않아도 괜찮아요” 문구를 유지한다.

### 개인정보에 민감한 사용자

예상 반응:

- 기록이 서버로 전송되는지 먼저 확인하려 할 수 있다.
- 로그인 없는 점을 장점으로 볼 수 있다.

위험:

- 개인정보처리방침 URL이나 문의 이메일이 비어 있으면 신뢰가 깨질 수 있다.
- 기록 내보내기/삭제가 눈에 잘 안 띄면 불안할 수 있다.

대응:

- Play Store 등록 전 실제 개인정보처리방침 URL을 연결한다.
- 앱 안에 “기록은 이 기기 안에 저장됩니다”를 명확히 둔다.
- 모든 기록 삭제와 내보내기를 돌아보기 화면에서 찾기 쉽게 유지한다.

### 스마트폰 사용이 익숙하지 않은 사용자

예상 반응:

- 하단 탭과 긴 스크롤을 동시에 이해하기 어려울 수 있다.
- 칩 버튼이 많으면 어떤 것이 선택됐는지 헷갈릴 수 있다.

위험:

- 선택 상태를 놓칠 수 있다.
- 지도 가로 스크롤을 발견하지 못할 수 있다.

대응:

- 선택된 칩의 대비를 충분히 높인다.
- 지도 위에 “가로로 밀어 더 볼 수 있습니다” 문구를 유지한다.
- 버튼 높이와 터치 영역을 Android 실제 기기에서 확인한다.

## 4. P0/P1/P2/P3 우선순위

### P0: Play Store 출시 전 반드시 해결

| 항목 | 이유 |
| --- | --- |
| Android 실제 기기에서 키보드와 하단 탭 충돌 확인 | 기록 입력이 앱의 첫 행동이므로 입력 경험이 깨지면 출시 불가 |
| 개인정보처리방침 URL 실제 연결 | Play Store 등록과 사용자 신뢰에 필수 |
| 문의 이메일 실제 값 연결 | 사용자 문의와 Play Store 등록에 필요 |
| 모든 기록 삭제 후 새로고침 유지 확인 | 개인 기록 앱의 신뢰 핵심 |
| AAB 업로드 인증서/서명 키 일관성 확인 | 잘못된 키로 업로드하면 배포 흐름이 막힘 |

### P1: 폐쇄 테스트 전 해결 권장

| 항목 | 이유 |
| --- | --- |
| 지도 Top 3 해석 문구 강화 | 사용자가 지도를 바로 이해해야 앱 가치가 전달됨 |
| 시도 카드 입력 부담 축소 | 핵심 루프 진입률을 높이기 위해 필요 |
| 돌아보기 화면의 다음 행동 강화 | 장기 사용 루프 형성에 필요 |
| 실제 Android 뒤로가기 동작 확인 | WebView 앱에서 자주 발생하는 사용성 문제 |
| 작은 화면 기기 QA | Play Store 사용자 기기 다양성 대응 |

| 항목 | 이유 |
| --- | --- |

## 6. 실제 사용자 5명 테스트 질문

테스트 방식은 1명당 10~15분 정도로 진행한다. 사용자가 앱을 처음 보는 상황을 유지하고, 설명을 최소화한 뒤 관찰한다.

### 공통 과제

1. 앱을 열고 무엇을 하는 앱인지 말해달라고 한다.
2. 예시 칩을 사용하거나 직접 입력해서 첫 기록을 남기게 한다.
3. 지도 화면에서 무엇이 보이는지 설명하게 한다.
4. 시도 카드를 하나 만들어보게 한다.
5. 돌아보기 화면에서 기록 관리와 개인정보 안내를 찾게 한다.

### 질문

1. 처음 화면만 보고 이 앱이 무엇을 하는 앱인지 이해됐나요?
2. 첫 기록을 남기는 과정에서 망설인 부분이 있었나요?
3. 지도 화면에서 “자주 막힌 위치”가 무엇을 뜻하는지 이해됐나요?
4. 시도 카드를 만드는 과정이 부담스럽거나 숙제처럼 느껴졌나요?
5. 기록이 이 기기 안에 저장된다는 안내가 충분히 신뢰감을 줬나요?

### 관찰 포인트

- 첫 기록까지 걸린 시간
- 예시 칩 사용 여부
- 분류 선택에서 멈춘 지점
- 지도 화면에서 Top 3를 먼저 보는지
- 시도 카드 입력 중 포기하는 필드
- 개인정보 안내를 스스로 찾는지

## 결론

v0.3.2 자동 QA 통과는 마찰지도가 기본 기능 흐름을 안정적으로 수행한다는 좋은 신호다. 하지만 Play Store에 올릴 앱으로 보려면 실제 사용자 5명 테스트와 Android 실제 기기 QA가 반드시 필요하다.

가장 중요한 리스크는 세 가지다.

1. 첫 기록 전에 사용자가 분류 선택을 부담스럽게 느끼는지
2. 지도 화면을 보고 “어디를 보면 되는지” 바로 이해하는지
3. 시도 카드와 돌아보기가 실제로 다음 행동으로 이어지는지

따라서 v0.3.3의 중심은 새 기능 확장이 아니라, 실제 사용자에게 더 가볍고 믿을 수 있는 흐름으로 다듬는 것이다.
