# 마찰지도 v0.3.1 모바일 화면 흐름 개선 계획

작성일: 2026-04-29  
브랜치: `v0.3.1-mobile-flow`  
목표: 긴 스크롤형 대시보드를 줄이고, 사용자가 핸드폰 한 화면 단위로 앱의 목적을 이해하게 만든다.

## 1. 문제 정의

현재 v0.3.0 화면은 기능이 한 페이지에 길게 쌓여 있습니다.

이 구조는 웹 대시보드에서는 자연스럽지만, 모바일 앱에서는 다음 문제가 생깁니다.

- 처음 사용자가 어디서 시작해야 할지 오래 훑어야 합니다.
- 기록, 지도, 실험, 설정이 한 화면 흐름 안에 섞여 보입니다.
- 아래로 계속 내려야 해서 앱보다 문서나 대시보드처럼 느껴질 수 있습니다.
- 마찰 지도와 작은 실험 카드가 첫 기록 경험을 밀어낼 수 있습니다.
- Play Store 스크린샷에서 각 화면의 목적이 선명하게 보이기 어렵습니다.

v0.3.1에서는 화면을 기능 단위가 아니라 사용자 이해 단위로 나눕니다.

## 2. 핵심 목표

사용자가 각 화면을 열었을 때 바로 이렇게 이해해야 합니다.

```text
남기기: 아, 막힌 순간을 적는 곳이구나.
보기: 아, 내가 자주 막히는 위치를 보는구나.
바꾸기: 아, 다음에 덜 버겁게 할 방법을 만드는구나.
관리: 아, 내 기록과 앱 정보를 관리하는 곳이구나.
```

## 3. 새 정보 구조

하단 탭 4개를 기준으로 화면을 나눕니다.

```text
남기기 | 보기 | 바꾸기 | 관리
```

각 탭은 한 가지 목적만 가집니다.

| 탭 | 목적 | 핵심 컴포넌트 |
| --- | --- | --- |
| 남기기 | 오늘 막힌 순간을 빠르게 기록 | `FrictionForm`, 최근 기록 1~2개 |
| 보기 | 자주 막힌 위치와 패턴 확인 | `FrictionMap`, `FrictionSummary`, `WeeklyReport` |
| 바꾸기 | 기록을 작게 바꿔보기 카드로 전환 | `ExperimentBuilder`, `ExperimentList` |
| 관리 | 기록 확인, 내보내기, 삭제, 개인정보 안내 | `FrictionList`, `SettingsPanel` |

## 4. 화면 1: 남기기

### 화면 목적

첫 화면입니다. 사용자는 여기서 45초 안에 첫 기록을 남길 수 있어야 합니다.

### 상단 문구

```text
오늘 어디서 막혔나요?
미뤘던 일, 부담스러운 답장, 시작하지 못한 순간을 한 줄로 남겨보세요.
```

### 화면 구성

- 앱 이름과 설정 진입 버튼
- 한 줄 기록 입력
- 빠른 예시 칩
- 그때 느낌
- 어느 쪽 일이었나요?
- 어디에서 막혔나요?
- 얼마나 버거웠나요?
- 기록하기 버튼
- 저장 직후 피드백
- 최근 기록 1~2개 미리보기

### 줄일 것

- 전체 기록 목록은 여기서 길게 보여주지 않습니다.
- 마찰 지도는 여기서 크게 보여주지 않습니다.
- 작은 실험 폼은 첫 화면에 노출하지 않습니다.

## 5. 화면 2: 보기

### 화면 목적

기록이 지도와 요약에 어떻게 반영되는지 확인하는 화면입니다.

### 상단 문구

```text
자주 막힌 위치
어느 쪽 일에서, 어느 순간에 자주 막히는지 볼 수 있어요.
```

### 화면 구성

- 기록 개수별 안내
- 자주 막힌 위치 Top 3
- 마찰 지도
- 반복 요약
- 주간 리포트

### 기록 개수별 처리

| 기록 수 | 화면 톤 |
| --- | --- |
| 0개 | 샘플 미리보기와 “실제 기록은 저장되지 않음” 안내 |
| 1개 | “관찰을 시작한 단계” 안내 |
| 2개 | “조금 더 쌓이면 위치가 선명해짐” 안내 |
| 3개 이상 | Top 3와 지도를 더 강조 |

## 6. 화면 3: 바꾸기

### 화면 목적

반복되는 막힘을 다음에 덜 버겁게 할 작은 방법으로 바꾸는 화면입니다.

### 상단 문구

```text
작게 바꿔보기
바로 해결하지 않아도 괜찮습니다. 다음에 덜 버겁게 할 작은 방법을 정해보세요.
```

### 화면 구성

- 기록이 없을 때: 먼저 기록이 필요하다는 빈 상태
- 기록이 1~2개일 때: 지금 해도 되고 더 모아도 된다는 안내
- 기록이 3개 이상일 때: 자주 막힌 위치에서 하나 골라보라는 안내
- `ExperimentBuilder`
- `ExperimentList`

### UX 원칙

- 숙제처럼 보이지 않게 합니다.
- “정답”, “성공/실패 판정” 같은 언어를 피합니다.
- “왜 그랬을까요?”, “다음에 작게 바꿔볼 것”, “어떻게 확인할까요?”처럼 풀어 씁니다.

## 7. 화면 4: 관리

### 화면 목적

사용자가 자신의 기록과 앱 정보를 통제하는 화면입니다.

### 상단 문구

```text
내 기록
기록은 이 기기 안에 저장됩니다.
```

### 화면 구성

- 전체 마찰 기록 목록
- 기록 삭제
- 기록 내보내기
- 모든 기록 삭제
- 개인정보 안내
- 앱 버전
- 문의 이메일 자리

### UX 원칙

- 데이터 통제권을 분명히 보여줍니다.
- 삭제 버튼은 조용하지만 명확하게 둡니다.
- 삭제 전 확인 단계는 유지합니다.

## 8. 하단 탭 디자인

### 탭 이름

```text
남기기
보기
바꾸기
관리
```

### 디자인 원칙

- 모바일 하단 고정
- 엄지로 누르기 쉬운 높이
- 현재 탭이 명확히 보이는 색상
- 다크모드 대비 유지
- 아이콘은 가능하면 단순한 문자/기호보다 lucide 아이콘 또는 기존 스타일에 맞는 SVG 사용

### 접근성

- 버튼에는 `aria-label`을 둡니다.
- 현재 탭에는 `aria-current="page"` 또는 동등한 표시를 둡니다.
- 포커스 스타일을 유지합니다.

## 9. 컴포넌트 분리 제안

새로 만들거나 정리할 수 있는 컴포넌트:

- `BottomNav`
- `RecordScreen`
- `MapScreen`
- `ExperimentScreen`
- `ManageScreen`

단, 첫 구현에서는 과한 리팩터링을 피하고 `page.tsx` 내부에서 탭 구조를 먼저 안정화한 뒤 분리해도 됩니다.

## 10. 기존 기능 유지 기준

v0.3.1은 정보 구조 개선이지 새 기능 확장이 아닙니다.

반드시 유지할 기능:

- 마찰 기록 추가
- 빠른 예시 칩 자동 선택값 채우기
- 빈 입력 방지
- localStorage 저장
- 새로고침 후 기록 유지
- 기록 삭제
- 지도 개수 정확성
- Top 3 정확성
- 반복 요약
- 작게 바꿔보기 카드 생성
- 상태 변경
- 실험 카드 삭제
- 연결된 기록이 삭제되어도 실험 카드 유지
- 다크모드
- 기록 내보내기
- 모든 기록 삭제

## 11. 구현 순서

1. `BottomNav` 컴포넌트를 만든다.
2. `page.tsx`에 `activeTab` 상태를 추가한다.
3. 기존 긴 레이아웃을 4개 화면 영역으로 나눈다.
4. 기본 탭은 `남기기`로 둔다.
5. `남기기` 화면은 첫 기록에 집중하도록 줄인다.
6. `보기` 화면으로 지도/요약/주간 리포트를 이동한다.
7. `바꾸기` 화면으로 실험 작성/목록을 이동한다.
8. `관리` 화면으로 전체 기록/설정/개인정보 안내를 이동한다.
9. 모바일 하단 고정 탭 여백을 확보한다.
10. 데스크톱에서는 하단 탭을 유지하거나 상단/사이드 탭처럼 보이게 조정한다.
11. `pnpm.cmd build`를 실행한다.
12. Android `cap sync`와 signed AAB 재생성 여부를 결정한다.

## 12. QA 체크리스트

### 기능

- [ ] 기록을 추가할 수 있다.
- [ ] 예시 칩이 텍스트와 기본 선택값을 함께 채운다.
- [ ] 저장 후 피드백이 보인다.
- [ ] 새로고침 후 기록이 유지된다.
- [ ] 보기 탭에서 지도 개수가 맞다.
- [ ] 보기 탭에서 Top 3가 맞다.
- [ ] 바꾸기 탭에서 실험 카드를 만들 수 있다.
- [ ] 실험 상태를 바꿀 수 있다.
- [ ] 관리 탭에서 기록을 삭제할 수 있다.
- [ ] 관리 탭에서 내보내기가 가능하다.
- [ ] 모든 기록 삭제가 상태와 저장소에 반영된다.

### 모바일

- [ ] 첫 화면에서 입력 경험이 바로 보인다.
- [ ] 하단 탭이 화면 밖으로 밀리지 않는다.
- [ ] 칩 버튼이 손가락으로 누르기 쉽다.
- [ ] 지도는 보기 탭 안에서 가로 스크롤이 된다.
- [ ] 실험 작성 폼이 화면 밖으로 튀지 않는다.
- [ ] 관리 화면의 삭제/내보내기 버튼이 명확하다.

### 문구

- [ ] “실패”, “의지 부족”, “문제 영역” 같은 표현이 불필요하게 보이지 않는다.
- [ ] 첫 화면은 “막힘” 중심으로 이해된다.
- [ ] 내부 개념은 유지하되 “가설/실험/관찰”이 너무 딱딱하게 노출되지 않는다.

## 13. Play Store 스크린샷 구성 제안

v0.3.1 이후에는 스크린샷을 화면 단위로 찍습니다.

1. 남기기: “오늘 어디서 막혔나요?”
2. 보기: “자주 막힌 위치”
3. 바꾸기: “작게 바꿔보기”
4. 관리: “기록은 이 기기 안에 저장됩니다”

이 구성이 되면 Play Store에서 앱의 흐름이 훨씬 선명해집니다.
