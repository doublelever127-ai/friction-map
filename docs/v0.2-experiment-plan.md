# 마찰지도 v0.2 작은 실험 카드 계획

## 1. v0.2 목표

v0.2의 목표는 기록된 마찰을 사용자가 직접 세운 원인 가설과 3일, 5일, 7일짜리 작은 실험으로 바꾸는 기능을 정식화하는 것입니다.

이 기능은 조언 생성기가 아닙니다. 앱이 정답을 말해주는 대신, 사용자가 자신의 기록을 바탕으로 가설을 적고 작은 실험을 설계하고 관찰하도록 돕습니다.

## 2. v0.1에서 v0.2로 확장되는 사용자 흐름

v0.1 흐름:

```text
마찰 기록 -> 분류 -> 기록 목록 -> 마찰 지도 -> 반복 요약
```

v0.2 확장 흐름:

```text
마찰 기록 선택 -> 원인 가설 작성 -> 실험 기간 선택 -> 작은 실험 작성 -> 성공 기준 작성 -> 예상대로 안 됐을 때의 해석 작성 -> 실험 카드 관리
```

사용자는 먼저 반복되는 마찰의 위치를 확인한 뒤, 특정 기록 하나를 골라 작게 확인해볼 실험으로 바꿉니다.

## 3. 작은 실험 카드 데이터 모델

권장 타입:

```ts
export type FrictionExperimentStatus =
  | "진행 전"
  | "진행 중"
  | "관찰 완료"
  | "보류";

export type FrictionExperiment = {
  id: string;
  frictionLogId: string;
  title: string;
  hypothesis: string;
  action: string;
  durationDays: 3 | 5 | 7;
  successCriteria: string;
  failureInterpretation: string;
  status: FrictionExperimentStatus;
  createdAt: string;
  updatedAt: string;
};
```

`failureInterpretation`은 실패를 의지 부족으로 해석하지 않도록, 실험 단위나 조건을 다시 살피는 문장으로 작성하게 합니다.

## 4. 현재 코드에 이미 있는 구조

현재 v0.2 프리뷰 구조로 아래 코드가 존재합니다.

- `src/types/friction.ts`: `FrictionExperiment`, `CreateFrictionExperimentInput`
- `src/lib/experimentStorage.ts`: `friction-map.experiments.v1` 저장 키와 load/save/create 함수
- `src/components/ExperimentBuilder.tsx`: 마찰 기록 선택, 가설, 실험, 성공 기준, 해석 입력 폼
- `src/components/ExperimentList.tsx`: 실험 카드 목록, 연결된 마찰 기록 표시, 삭제 버튼

v0.1 범위 정리를 위해 현재 `src/app/page.tsx`에서는 작은 실험 카드 섹션을 노출하지 않습니다.

## 5. 재사용할 코드

- `ExperimentBuilder`의 기본 입력 필드 구성
- `ExperimentList`의 카드 렌더링 구조와 연결된 마찰 기록 표시 방식
- `experimentStorage`의 브라우저 환경 안전 처리
- `friction-map.experiments.v1` localStorage 키
- 기존 `FrictionLog`와 연결하는 `frictionLogId` 구조

## 6. 수정이 필요한 코드

- `FrictionExperiment`에 `status`, `updatedAt`을 추가합니다.
- `durationDays`를 `number`보다 좁은 `3 | 5 | 7` 타입으로 제한합니다.
- `CreateFrictionExperimentInput`에서 `status` 기본값을 어떻게 다룰지 정합니다. 기본값은 `"진행 전"`이 적절합니다.
- `experimentStorage`의 유효성 검사에 `status`, `updatedAt`, 기간 타입 검사를 추가합니다.
- 생성 함수가 저장까지 담당할지, 객체 생성만 담당하고 페이지에서 저장할지 정리합니다. v0.2에서는 상태와 저장 흐름을 한곳에서 읽기 쉽게 관리하는 쪽이 좋습니다.
- `page.tsx`에 작은 실험 섹션을 다시 연결하되 v0.1 핵심 화면을 밀어내지 않는 위치에 배치합니다.
- 삭제 외에 상태 변경을 지원하는 콜백을 추가합니다.

## 7. 필요한 localStorage 키

기존 키를 유지합니다.

| 데이터 | 저장 키 |
| --- | --- |
| 작은 실험 카드 | `friction-map.experiments.v1` |

기존 프리뷰 데이터가 있을 수 있으므로 `status`와 `updatedAt`이 없는 항목은 불러올 때 기본값을 보완하는 마이그레이션을 고려합니다.

## 8. 필요한 컴포넌트

- `ExperimentBuilder`: 새 실험 작성 폼
- `ExperimentList`: 실험 카드 목록
- `ExperimentStatusControl`: 실험 상태를 바꾸는 작은 컨트롤
- `ExperimentCard`: 목록 내부 카드가 커질 경우 분리할 카드 컴포넌트
- `ExperimentEmptyState`: 기록이 없거나 실험이 없을 때의 부드러운 빈 상태

초기 구현에서는 새 컴포넌트를 과하게 늘리지 않고, `ExperimentBuilder`와 `ExperimentList`를 먼저 정식화합니다.

## 9. 화면 배치 제안

v0.2에서도 첫 화면의 중심은 마찰 기록과 지도입니다. 작은 실험 카드는 기록과 패턴을 본 뒤 이어지는 영역으로 배치합니다.

권장 순서:

1. 헤더
2. 마찰 기록 입력과 마찰 지도
3. 반복 요약과 주간 리포트
4. 작은 실험 작성
5. 실험 카드 목록
6. 마찰 기록 목록

모바일에서는 모두 한 열로 쌓고, 데스크톱에서는 작은 실험 작성과 목록을 2열로 둘 수 있습니다.

## 10. 사용자 입력 필드

- 연결할 마찰 기록
- 실험 이름
- 원인 가설
- 3일, 5일, 7일 중 실험 기간
- 3~7일 동안 해볼 작은 실험
- 성공 기준
- 예상대로 안 됐을 때의 해석

필수 입력값은 공백을 `trim()`으로 막고, 사용자를 압박하는 오류 문구 대신 조용한 안내를 사용합니다.

## 11. 실험 상태 관리 방식

상태 후보:

- `진행 전`
- `진행 중`
- `관찰 완료`
- `보류`

상태 변경은 점수나 streak가 아니라 현재 관찰 위치를 표시하는 용도로만 사용합니다. 상태가 바뀔 때마다 `updatedAt`을 갱신합니다.

## 12. 삭제/수정/완료 기능 범위

v0.2에 넣을 기능:

- 실험 생성
- 실험 목록 표시
- 실험 상태 변경
- 실험 삭제

v0.2에서 선택적으로 검토할 기능:

- 실험 수정

수정 기능은 폼 복잡도를 키우므로 v0.2.0에서는 제외하고 v0.2.1 후보로 둘 수 있습니다. 완료는 별도 성공/실패 판정이 아니라 `관찰 완료` 상태로 표현합니다.

## 13. v0.2에서 일부러 넣지 않을 기능

- AI 조언 생성
- 실험 성공률 점수화
- streak
- 랭킹이나 공유 기능
- 과한 알림
- 서버 저장, 로그인, 데이터베이스
- 자동 진단 문장
- 정답처럼 보이는 추천 실험

## 14. 구현 순서

1. `FrictionExperimentStatus`, `status`, `updatedAt` 타입 추가
2. `experimentStorage` 유효성 검사와 기본값 보완
3. `ExperimentBuilder`를 정식 입력 폼으로 정리
4. `page.tsx`에 실험 상태와 저장 흐름 연결
5. `ExperimentList`에서 상태 표시와 상태 변경 컨트롤 추가
6. 삭제 기능 유지 확인
7. 빈 상태와 문구 점검
8. 모바일 레이아웃 확인
9. `pnpm build` 실행
10. v0.2 테스트 체크리스트 작성

## 15. 테스트 체크리스트

- `pnpm build`가 성공합니다.
- 마찰 기록이 없을 때 실험 작성 폼이 부드러운 빈 상태를 보여줍니다.
- 마찰 기록을 선택해 실험 카드를 만들 수 있습니다.
- 공백 입력은 저장되지 않습니다.
- 실험 기간은 3일, 5일, 7일 중 하나만 저장됩니다.
- 새 실험의 기본 상태는 `진행 전`입니다.
- 상태를 바꾸면 화면과 `localStorage`에 함께 반영됩니다.
- 새로고침 후 실험 카드와 상태가 유지됩니다.
- 실험 삭제 후 새로고침해도 다시 나타나지 않습니다.
- 연결된 마찰 기록 내용이 실험 카드에 표시됩니다.
- 연결된 마찰 기록이 삭제된 경우에도 실험 카드가 깨지지 않습니다.
- 모바일에서 실험 작성 폼과 카드 목록이 화면 밖으로 튀지 않습니다.
- 문구가 사용자를 비난하거나 진단하지 않습니다.
